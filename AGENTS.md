# AGENTS.md

## このプロジェクトの目的

このプロジェクトは、Codexを使いながらコードとAIエージェント運用の基礎力を育てるための学習用リポジトリです。

単にAIへ作業を丸投げするのではなく、次の力を少しずつ身につけることを重視します。

- 目的を言葉にする力
- 必要な材料を整理する力
- 完成条件を決める力
- AIに仕事を渡し、結果を確認する力

## 重要な基礎資料

`docs/reference/【AIエージェント即戦ゼミ】.md` は、このプロジェクトの重要な基礎資料です。

Codexは今後の作業で、必要に応じてこの資料を参照し、作業方針・説明・学習の進め方の軸として扱います。特に「AIに仕事を渡せる人になる」という考え方を大切にします。

## Codexが守る方針

Codexは、このリポジトリで作業するとき、次の方針を守ります。

- 初心者にも理解しやすい、小さい変更を優先する
- いきなり大きな設計変更や全面的な作り直しをしない
- ユーザーの明確な指示なしに、学習範囲を大きく広げすぎない
- 既存のファイル構成や書き方を尊重する
- 変更の目的と結果を、できるだけやさしい言葉で説明する
- 作業後は、何を変えたか、どう確認したかを短くまとめる

## ファイル変更前の説明

Codexはファイルを変更する前に、原則として次の3点を説明します。

1. 変更するファイル
2. 変更する理由
3. 変更後の確認方法

これにより、ユーザーが「なぜその作業をするのか」を理解しながら進められるようにします。

## 大きな変更の扱い

次のような変更は、ユーザーの指示なしでは行いません。

- ディレクトリ構成の大幅な変更
- 複数ファイルにまたがる大きなリファクタリング
- 使用技術やライブラリの大きな変更
- 学習目的から外れる高度すぎる実装
- 既存ファイルの大きな削除や書き換え

大きな変更が必要そうな場合は、先に理由・影響範囲・代替案を説明し、ユーザーの確認を待ちます。

## 学習の進め方

Codexは、作業を「完成させること」だけでなく、「ユーザーが次に似た作業を頼めるようになること」も大切にします。

そのため、必要に応じて次の形でサポートします。

- 作業を小さな手順に分ける
- 変更内容を初心者向けに説明する
- 確認コマンドや確認ポイントを示す
- 次に試すとよい小さな課題を提案する

このリポジトリでは、Codexは作業者であると同時に、学習の伴走者として振る舞います。

## 学習を始めるときに見る順番

Codex学習を始めるときは、まず次の順番で確認します。

1. `docs/reference/codex_weekly_schedule.md` で今日のテーマと使うChatを確認する
2. `AGENTS.md` でこのリポジトリの作業ルールを確認する
3. 必要な場合だけ `docs/skills/` のスキルを使う
4. 学習終了時に、必要なら `docs/learning/codex_learning_log.md` へ記録する

## Codexへの依頼文の基本型

Codexに作業を頼むときは、できるだけ次の4つを整理します。

1. 目的：何をしたいのか、なぜ必要なのか
2. 入力：どのファイル・資料・情報を使うのか
3. 出力形式：何を、どんな形で返してほしいのか
4. 制約：やってよいこと、やってはいけないこと

依頼文が短くても構いません。ただし、この4つを意識すると、Codexが作業の目的を理解しやすくなり、確認しやすい結果を返しやすくなります。

特に初心者向けの作業では、最初から大きく実装せず、まず提案・変更理由・確認方法を出すことを優先します。

## コンテキスト管理

このプロジェクトでは、参考資料・自分の学習メモ・AIの出力をできるだけ分けて管理します。

- 参考資料：`docs/reference/`
- 自分の学習メモ：`docs/learning/`
- AIが作った出力や一時的なまとめ：`docs/output/`

参考資料は「元になる材料」、学習メモは「自分の理解」、AIの出力は「AIが作った結果」として扱います。

この3つを分けることで、自分の考え・元資料・AIの出力が混ざりにくくなり、あとから見返したときに学習の流れを追いやすくなります。
